## What just shipped

**Plan A — TimeCapsule Registry Foundation** *(now live on `main`)*

A sealed, GPG-signed, mount-based image store on the +1 disk layer. Podman reads it via `additionalimagestores`. No more reaching out to docker.io / ghcr.io at runtime. Every image S7 needs to run will live here, one signed tar at a time, indexed by a single `manifest.json`.

What's in it:
- The directory layout at `/s7/timecapsule/registry/`
- A JSON-Schema-validated manifest with atomic Python writer (16 unit tests)
- A boot-time verify script with mocked **and** real-GPG round-trip tests (5 tests)
- A storage.conf snippet that wires podman into the additional store
- A systemd unit (rootless-friendly) that runs the verify before podman starts
- A modified intake adapter that promotes new images by `podman save` → `gpg --detach-sign --armor` → manifest entry — instead of dumping into the live graphroot

It is the first piece of S7's answer to "what happens when an upstream registry deletes an image we depend on." The answer is: nothing. We have it. We signed it. It boots from us.

**Plan B0 — Boot-Server Validator** *(also live on `main`)*

A small bash + pytest unit that takes a freshly-built S7 base image, boots it as a podman container, waits for it to actually serve, runs HTTP smoke checks, tears down. Wired into the lifecycle test as gate `B01`. Future plans get the boot-server gate "for free."

The lifecycle test count went from 39 to 40. The new test passes today by design — it SKIPs because the dev box's port 8080 is occupied by Tonya's running SPA, and the fresh `s7-fedora-base:v2026.04.13` image isn't running a SPA inside itself yet (that's Plan B work). On a clean production box, B01 will actually boot, smoke-test, and tear down.

**Plan B (safe half) — TimeCapsule populated, network restored, scan integrated** *(also live on `main`)*

The infrastructure for the eventual service migration. **No service touched tonight. Tonya's QUBi keeps running on its current containers.** What landed:

- Rootless `storage.conf` installed at `/s7/.config/containers/storage.conf` — podman now sees the TimeCapsule additional store
- The verify script + systemd USER unit installed at `/s7/.local/bin/` and `/s7/.config/systemd/user/` — runs cleanly, exit 0
- The `qubi` podman network recreated at `172.16.7.32/27` (shifted from the brand-canonical `.0/27` because a 2-day-old removed network left a `172.16.7.1/32` orphan on `lo` we couldn't clear without sudo — flagged as exactly the kind of "UNKNOWN" environmental drift the discipline is designed to protect against)
- 6 service upstream refs pinned in `iac/manifest.yaml` (cyberchef, jellyfin, nomad, qdrant, mysql, redis)
- **Trivy v0.69.3 installed and integrated** — scanned every candidate image plus the bootc base
- **Real CVE finding documented**: *42 CRITICAL + 443 HIGH across the 6 third-party images we currently run*. Zero in the S7-built bootc base.

That last bullet is the most important sentence on this page. **The work S7 actually controls is at zero.** Every CVE is in something we inherited from upstream. The intake-gate strategy is *validated by this finding*: signing dirty bits with the S7 brand without scanning them first is a regression, not a win, so we wiped the unscanned signed tars and committed the scan reports as the audit trail instead.

The image-hardening plan that comes from this finding is its own next plan: nomad with 32 CRITICALs is the headliner, mysql/redis/qdrant need fresh upstream pins, cyberchef + jellyfin are tolerable with an upgrade later. The numbers are knowable now because we built the scanner pipeline tonight.

## What got documented

- **Two-tier release discipline + public freeze.** A new `lifecycle` branch in private holds active devops. Promotion to private `main` is "go-live-private." Promotion to public `main` is "go-live-public" and **happens only on Core Update days**. Next: **2026-07-07 07:00 CT**. Between now and then, public is frozen. Dependabot only.
- **R03 removed from the lifecycle test.** It was checking "private and public are in sync" — which is a guaranteed-fail under the freeze rule. The sync semantics moved to a discrete `iac/promote-to-public.sh` gate that runs only on Core Update days.
- **The cube/desktop rule.** Every change inside the 6561-cell trinity cube is safe to iterate. Every change to the desktop layer (Budgie, launchers, wallpaper) is sacred and follows a write-barrier discipline. One desktop break = one lost household.
- **The "no UNKNOWNs" discipline.** Environmental drift is a real source of failure, not just code drift. Tonight's `172.16.7.1` lo orphan is a textbook example: removed two days ago at the network layer, the IP alias survived, and broke a planned subnet creation. The rule going forward: protect against UNKNOWNs as actively as scope drift.
