## Why v2.7 exists — the two /discern bugs

The 2026-04-13 `/witness` ConnectError debugging pass surfaced a second
class of bug in the engine code that lives inside the admin image. The
engine code is NOT bind-mounted from the host — it's baked into the
image at `/app/engine/`. That means any code change to engine/\*.py
requires a new image to take effect. `/witness` had two issues
(ConnectError + foreign-key violation) both of which were fixed
without a rebuild; `/discern` has TWO issues that cannot be fixed
without one.

### Bug 1 — `DiscernRequest.session_id` is `str`, not `uuid.UUID`

**Symptom:** `POST /discern {"session_id":"dbg1","tokens":["a","b"]}` →
500 Internal Server Error with stack trace:

```
File "/app/engine/s7_discernment.py", line 70, in run_discernment
psycopg2.errors.InvalidTextRepresentation:
    invalid input syntax for type uuid: "dbg1"
```

**Root cause:** FastAPI accepts any string for `session_id` (because the
Pydantic model types it as `str`), passes it to `run_discernment()`,
which passes it to SQL `%s::uuid` cast, which crashes with a
psycopg2 error that surfaces as a 500.

**Fix (committed 2026-04-13 in `e0fb251`):** `engine/s7_server.py` —
change `session_id: str` → `session_id: uuid.UUID` on all five request
models (`DiscernRequest`, `StoreRequest`, `BreakerRequest`,
`WitnessRequest`, `QuantiRequest`). Pydantic rejects non-UUID input at
the validator layer with a 422 + structured error. The `sid =
req.session_id or str(uuid.uuid4())` lines in the `/witness` and
`/quanti` handlers are also fixed to coerce `uuid.UUID | None` to
`str` for SQL parameter passing.

**Deploy status:** in git only, NOT live. Lives inside the admin image.

### Bug 2 — `run_discernment()` doesn't INSERT a session row first

**Symptom:** `POST /discern {"session_id":"<valid UUID>","tokens":["a","b"]}`
→ 500 Internal Server Error with stack trace:

```
File "/app/engine/s7_discernment.py", line 70, in run_discernment
psycopg2.errors.ForeignKeyViolation:
    insert or update on table "passes"
    violates foreign key constraint "passes_session_id_fkey"
```

**Root cause:** `cws_inference.passes` has a foreign key to
`cws_inference.sessions`. `run_discernment()` INSERTs directly into
`passes` without first creating a session row, so if the session_id
is new, the FK fails.

For comparison, `run_witness()` does this right — line 31-37 of
`engine/s7_witness.py`:

```python
cur.execute("""
    INSERT INTO cws_inference.sessions
        (id, input_tokens, input_constant, input_location,
         scale_weight, pulse_step, inference_path, started_at)
    VALUES (%s::uuid, %s, 1.0, 0.0, 1.0, 0,
            'standard'::cws_routing.inference_path, NOW())
    ON CONFLICT (id) DO NOTHING
""", (session_id, input_tokens))
```

**Fix (NOT yet committed — defer to v2.7 commit):** add the same
`INSERT INTO cws_inference.sessions ... ON CONFLICT DO NOTHING`
preamble to `run_discernment()` before any INSERT into `passes` or
`discernment`. Copy the shape from `run_witness()` for consistency.

Specifically, in `engine/s7_discernment.py:63` (`run_discernment`),
insert the session-ensure block BEFORE the `for r in results:` loop:

```python
async def run_discernment(db_conn, session_id, tokens: list[str]) -> list[dict]:
    forward_scores = await score_tokens_forward(tokens)
    reverse_scores = await score_tokens_reverse(tokens)
    results = compute_discernment(tokens, forward_scores, reverse_scores)
    with db_conn.cursor() as cur:
        # v2.7 fix: ensure session row exists before inserting passes.
        # Before, FK on cws_inference.passes.session_id crashed when a
        # new session_id was passed.
        cur.execute("""
            INSERT INTO cws_inference.sessions
                (id, input_tokens, input_constant, input_location,
                 scale_weight, pulse_step, inference_path, started_at)
            VALUES (%s::uuid, %s, 1.0, 0.0, 1.0, 0,
                    'standard'::cws_routing.inference_path, NOW())
            ON CONFLICT (id) DO NOTHING
        """, (str(session_id), tokens))
        for r in results:
            ...  # existing INSERT into passes + discernment stays unchanged
```

**Deploy status:** not yet committed. Commit as part of the v2.7
pre-build review, alongside the Pydantic type fix.

## Rebuild mechanism — options ranked

### Option A — `podman commit` patch release (quick, pragmatic)

**Mechanism:**
1. Create a work container from `v2.6` with a no-op entrypoint:
   ```
   podman create --name admin-v27-work --entrypoint /bin/sleep \
       localhost/s7-skyqubi-admin:v2.6 infinity
   podman start admin-v27-work
   ```
2. Copy the fixed engine code in:
   ```
   podman cp /s7/skyqubi-private/engine/. admin-v27-work:/app/engine/
   ```
3. Commit as v2.7:
   ```
   podman commit admin-v27-work localhost/s7-skyqubi-admin:v2.7
   ```
4. Save as tarball for the sovereign-distribution path:
   ```
   podman save -o s7-skyqubi-admin-v2.7.tar \
       localhost/s7-skyqubi-admin:v2.7
   ssh-keygen -Y sign -f /s7/.ssh/s7-image-signing \
       -n file s7-skyqubi-admin-v2.7.tar
   ```
5. Update `skyqubi-pod.yaml` from `:v2.6` to `:v2.7`.
6. Pod restart via `fix-pod.sh` or `start-pod.sh`.
7. Run lifecycle → expect 53/53 still, PLUS /discern now works.
8. Add `E07 /discern full verdict` test.

**Pros:** 5-minute build, inherits v2.6's layers, only diff is
`/app/engine/`, easy to diff + audit.

**Cons:** Breaks the "immutable images from source" best practice.
Layer history is opaque — no Containerfile the rebuild is from.
Not suitable for long-term production but fine for this-box patch.

**Risk:** Low. v2.6 → v2.7 diff is a directory copy inside a
sleep container. Rollback is `podman rm v2.7 && revert pod yaml`.

### Option B — Full Containerfile rebuild from `iac/admin/Containerfile.draft`

**Mechanism:**
1. Build `s7-fedora-base` via `iac/build-s7-base.sh` (prerequisite).
2. Build admin image via `iac/admin/Containerfile.draft`:
   ```
   podman build -t localhost/s7-skyqubi-admin:v2.7 \
       -f iac/admin/Containerfile.draft \
       --build-arg VERSION=2.7 \
       --build-arg BUILD_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ) \
       --build-arg VCS_REF=$(git rev-parse HEAD) \
       .
   ```
3. Same steps 4–8 as Option A.

**Pros:** Real build from source. Auditable Containerfile. Works
for any future base image changes. Long-term correct.

**Cons:** Draft Containerfile has 6 known gaps (NodeSource install,
libvips naming, GraphicsMagick package name, entrypoint path,
PGDG/pgvector compat). Building it will fail in at least 2-3 places
that need hand-resolution. 1-3 hour effort, possibly more.

**Risk:** Medium. Untested. First build will almost certainly fail
and need iteration. Not for a quick patch.

### Option C — Defer to the finish-line plan's subsystem #4

**Mechanism:** Wait for subsystem #4 (admin container rebase) from
the 2026-04-13 finish-line plan to complete, which includes a
proper Containerfile port. That block has not started.

**Pros:** Architecturally correct. No duplicated work.

**Cons:** Indefinite timeline. `/discern` stays broken in the
meantime. `E06` covers `/witness` but `/discern` remains a known
gap.

### Recommendation (REVISED 2026-04-13 per Jamie's direction)

Jamie's direction: **"QUBI deploy without needing the base, but
prepared for F44."** This changes Option B's viability. Option B's
draft Containerfile is `FROM localhost/s7-fedora-base:latest`, which
means a v2.7 build requires `s7-fedora-base` to be built first. That
in turn means `install.sh` on a fresh F44 box would need to build
`s7-fedora-base` AS A PREREQUISITE before it could build/pull the
admin image. That violates the "deploy without needing the base"
principle — the user shouldn't have to build a base layer before
getting a working QUBi.

Revised recommendation:

**For this block (2026-04-13 private-updates-only):** NEITHER Option A
nor Option B runs. All fixes stay in git. The runtime is frozen per
Jamie's direction. This doc IS the deliverable for this block.

**For the interim patch (between now and GOLIVE):** **Option A
(podman commit)** — takes the currently-working v2.6 as the layer
base (since v2.6 IS a known-good admin image) and patches in the
engine fixes. No dependency on `s7-fedora-base`. No requirement to
build a base layer first. Preserves the "deploy without base"
principle.

**For GOLIVE Release 7 (2026-07-07):** A new Option D — a
Containerfile that's `FROM upstream-fedora-44` directly (not from
`s7-fedora-base`), so install.sh can build it on any F44 box
without a base-layer dependency. Inherits upstream Fedora's
maintenance + security updates naturally. This is the path
forward for GOLIVE; Option B's s7-fedora-base rebase is a
separate architectural track that can continue in parallel but
is NOT a prerequisite for GOLIVE.

### Option D (new — recommended for GOLIVE)

```dockerfile
# Direct from upstream Fedora 44 — no s7-fedora-base dependency.
# Respects the 'QUBI deploy without needing the base' principle.
FROM registry.fedoraproject.org/fedora:44 AS base

RUN dnf install -y --setopt=install_weak_deps=0 --setopt=tsflags=nodocs \
        nodejs nodejs-npm \
        GraphicsMagick vips-devel \
        gcc gcc-c++ make git \
        python3-devel postgresql-devel \
        python3-psycopg2 python3-httpx python3-fastapi python3-uvicorn \
        python3-pyyaml \
    && dnf clean all

# ... rest matches Option B from here down
```

Key differences from the `iac/admin/Containerfile.draft` (Option B):

- `FROM registry.fedoraproject.org/fedora:44` (not
  `localhost/s7-fedora-base:latest`)
- `dnf install` instead of `microdnf install` (full fedora:44 has
  dnf; fedora-minimal:44 has microdnf — our draft uses the minimal
  path which is unnecessarily constrained)
- Node.js from Fedora's default nodejs package (no NodeSource rpm)
- pip3 dependencies replaced with distro python3-* packages where
  available — fewer pip installs, more reproducible builds

Net: Option D ships without the `s7-fedora-base` prerequisite and
works with a single `podman build` against any box that can
`dnf install` from Fedora mirrors.

## Pre-build review checklist

Before any v2.7 build attempt:

1. Re-run the 53-test lifecycle suite on `lifecycle` branch. Target:
   53/53 or 54/54 (if E07 was added before the rebuild).
2. Verify `engine/s7_server.py` on HEAD has the Pydantic UUID fix.
3. Verify `engine/s7_discernment.py` on HEAD has the
   `INSERT INTO sessions ... ON CONFLICT DO NOTHING` preamble added.
4. Verify `engine/sql/s7-witness-personas-seed.sql` is present —
   it's bind-mounted into postgres init so it auto-applies on
   fresh initdb. Not baked into the admin image, but the v2.7 test
   plan includes a verification that the seed is reachable.
5. Verify `skyqubi-pod.yaml` has the `OLLAMA_URL` and `BITNET_URL`
   env vars in the admin container's env block (committed already
   in `e0fb251`). v2.7 deploy reuses the same yaml.
6. Verify `/s7/.env.secrets` still has all required values.
7. Verify `fix-pod.sh` is current (for rollback use during the
   build window).
8. Verify `install/preflight.sh` passes on the current box.
9. Have a rollback command ready:
   ```
   podman tag localhost/s7-skyqubi-admin:v2.7 \
              localhost/s7-skyqubi-admin:v2.7-rollback-from
   sed -i 's|:v2.7|:v2.6|' /s7/skyqubi-private/skyqubi-pod.yaml
   ```

## Rollback path

If v2.7 deploys but fails verification:

1. Edit `skyqubi-pod.yaml` back to `:v2.6`.
2. `podman pod rm -f s7-skyqubi`
3. `bash start-pod.sh`
4. `bash s7-lifecycle-test.sh` — confirm 53/53 back.
5. Don't delete the v2.7 image — it's on disk as evidence. Tag it
   as `:v2.7-rollback-from-$(date +%Y%m%d)` for audit.

The rollback is fast (~3 minutes) and known-good because v2.6 has
been running successfully on this box for a month.
