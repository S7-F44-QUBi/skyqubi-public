## Step 1 — Tag the current v2.6 for rollback safety

```bash
podman tag localhost/s7-skyqubi-admin:v2.6 \
           localhost/s7-skyqubi-admin:v2.6-pre-v27-rebuild
```

This creates a named rollback point that survives `podman image prune`
and is independent of tag juggling.

**Verification:** `podman images | grep v2.6-pre-v27-rebuild` should
show the new tag.

---

## Step 2 — Create the work container from v2.6

```bash
podman create --name admin-v27-work \
    --entrypoint /bin/sleep \
    localhost/s7-skyqubi-admin:v2.6 infinity

podman start admin-v27-work
```

The work container runs `/bin/sleep infinity` so the container stays
alive long enough for us to copy files into it.

**Verification:**
```bash
podman ps --filter name=admin-v27-work --format '{{.Names}} {{.Status}}'
# Expected: admin-v27-work Up N seconds
```

---

## Step 3 — Copy the fixed engine code into the work container

```bash
podman cp /s7/skyqubi-private/engine/. admin-v27-work:/app/engine/
```

`podman cp` overlays the host's `engine/` tree onto `/app/engine/`
inside the work container. This replaces the old engine code from v2.6
with the current HEAD of the lifecycle branch — which includes:

- `s7_server.py` (Pydantic UUID types)
- `s7_discernment.py` (session INSERT preamble)
- `s7_quanti.py` (session INSERT preamble)
- `s7_breaker.py` (consensus_session INSERT preamble)
- Plus all other engine files as-is

**Verification:**
```bash
# Spot-check: the Pydantic fix should be visible in the copied file
podman exec admin-v27-work grep -c "session_id: uuid.UUID" /app/engine/s7_server.py
# Expected: 4 (DiscernRequest, StoreRequest, BreakerRequest, and
# the uuid.UUID | None variant in WitnessRequest + QuantiRequest)

# Spot-check: run_discernment has the session INSERT preamble
podman exec admin-v27-work grep -c "INSERT INTO cws_inference.sessions" /app/engine/s7_discernment.py
# Expected: 1

# Spot-check: run_quanti has the same fix
podman exec admin-v27-work grep -c "INSERT INTO cws_inference.sessions" /app/engine/s7_quanti.py
# Expected: 1
```

## Step 4 — Commit the work container as v2.7

```bash
CREATED_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ)
GIT_SHA=$(cd /s7/skyqubi-private && git rev-parse --short HEAD)

podman commit \
    --message "v2.7 engine fixes (Pydantic UUID + session INSERT preambles) — patched from v2.6 via podman commit, git ${GIT_SHA}" \
    --change "LABEL org.opencontainers.image.version=2.7" \
    --change "LABEL org.opencontainers.image.created=${CREATED_DATE}" \
    --change "LABEL org.opencontainers.image.revision=${GIT_SHA}" \
    --change "LABEL org.s7.build-method=podman-commit-from-v2.6" \
    --change "LABEL org.s7.rebuild-date=2026-04-13" \
    admin-v27-work \
    localhost/s7-skyqubi-admin:v2.7
```

**Verification:**
```bash
podman images --format '{{.Repository}}:{{.Tag}} {{.ID}} {{.Size}}' \
    | grep s7-skyqubi-admin
# Expected output includes:
#   localhost/s7-skyqubi-admin:v2.7                <id> <size>
#   localhost/s7-skyqubi-admin:v2.6                <id> <same size>
#   localhost/s7-skyqubi-admin:v2.6-pre-v27-rebuild <id> <same size>
```

## Step 5 — Clean up the work container

```bash
podman stop admin-v27-work
podman rm admin-v27-work
```

The work container is no longer needed — the v2.7 image has the
changes baked in.

**Verification:** `podman ps -a | grep admin-v27-work` should return
nothing.

## Step 6 — Save v2.7 as a signed tarball (sovereign distribution)

```bash
TAR_DIR=/s7/skyqubi-private
cd "$TAR_DIR"

podman save \
    -o s7-skyqubi-admin-v2.7.tar \
    localhost/s7-skyqubi-admin:v2.7

# Sign with the S7 image key (matches v2.6's signing pattern)
if [ -f /s7/.ssh/s7-image-signing ]; then
    ssh-keygen -Y sign \
        -f /s7/.ssh/s7-image-signing \
        -n file \
        s7-skyqubi-admin-v2.7.tar
    ok "Tarball signed: s7-skyqubi-admin-v2.7.tar.sig"
else
    warn "Signing key not found at /s7/.ssh/s7-image-signing — skip sign"
fi

ls -lh s7-skyqubi-admin-v2.7.tar*
# Expected: ~350 MB tarball + ~250 byte signature
```

**Purpose:** air-gap delivery + intake-gate verification on other
boxes. Not strictly required for the local deploy but matches the
v2.6 pattern and keeps the sovereign distribution chain clean.

## Step 7 — Flip the pod yaml to reference v2.7

```bash
cd /s7/skyqubi-private
sed -i 's|localhost/s7-skyqubi-admin:v2.6|localhost/s7-skyqubi-admin:v2.7|g' \
    skyqubi-pod.yaml

# Verify the swap
grep -n "s7-skyqubi-admin" skyqubi-pod.yaml
# Expected: all instances show :v2.7
```

**Commit the yaml change** on the lifecycle branch:

```bash
git add skyqubi-pod.yaml
git commit -m "deploy(v2.7): flip admin image tag in pod yaml

Deploys the v2.7 engine fixes:
- Pydantic UUID types on DiscernRequest/StoreRequest/etc
- run_discernment session INSERT preamble
- run_quanti session INSERT preamble
- check_breaker consensus_session INSERT preamble

Rollback: sed -i 's|:v2.7|:v2.6|' skyqubi-pod.yaml && restart pod"
```

## Step 8 — Restart the pod to pick up v2.7

```bash

    bash /s7/skyqubi-private/start-pod.sh 2>&1 | tail -30
```

## Rollback — if verification fails

If Step 9's verification catches a problem, roll back immediately:

```bash
# 1. Flip pod yaml back to v2.6
sed -i 's|localhost/s7-skyqubi-admin:v2.7|localhost/s7-skyqubi-admin:v2.6|g' \
    /s7/skyqubi-private/skyqubi-pod.yaml

# 2. Remove the failing pod
podman pod rm -f s7-skyqubi

# 3. Restart with v2.6
sudo -u s7 -H env SKYQUBI_SECRETS=/s7/.env.secrets \
    bash /s7/skyqubi-private/start-pod.sh 2>&1 | tail -30

# 4. Verify back to known-good state
bash /s7/skyqubi-private/s7-lifecycle-test.sh 2>&1 | tail -3
# Expected: 53/53 PASS — LIFECYCLE VERIFIED

# 5. Commit the rollback so git is honest about what happened
cd /s7/skyqubi-private
git checkout -- skyqubi-pod.yaml  # OR git add + commit with rollback message
git commit -am "rollback(v2.7): flip back to v2.6 after verification failure

Triggered by: [describe the specific Step 9 check that failed]
v2.7 image still exists locally as localhost/s7-skyqubi-admin:v2.7
for diagnosis. Known-good rollback restored."
```

v2.6-pre-v27-rebuild is still on disk (Step 1). That's the deeper
rollback if the v2.6 tag itself gets corrupted somehow.
