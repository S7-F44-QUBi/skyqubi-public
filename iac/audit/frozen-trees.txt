# S7 Frozen Tree Pins — the canonical commit shas for tier-crossing branches.
#
# Audit gate zero #10 (Axis A — Drift) reads this file and compares each
# pinned ref's actual current sha against the pinned sha. A mismatch means
# the branch has moved without going through the Core Update gate.
#
# ## Format
#
#   <ref-spec>  <commit-sha-or-PENDING>  [# comment]
#
# ref-spec values:
#   private/lifecycle   — the active devops branch on the private repo
#   private/main        — the go-live-private branch on the private repo
#   public/main         — the public release branch (read from local mirror
#                         at /s7/skyqubi-public, NOT git ls-remote — the
#                         gate is offline-only by invariant of zeros 1-9)
#
# sha values:
#   <40-char hex>       — the canonical sha. Mismatch → BLOCK (sync refused).
#   PENDING             — known-unsettled state. The gate looks for a
#                         pinned.yaml entry with id `frozen-tree-<ref>-pending`
#                         and converts the finding from BLOCK to PINNED if
#                         found. If no pinned.yaml entry exists, it BLOCKS.
#
# Lines starting with # and blank lines are ignored.
#
# ## Resolution path
#
# Updating a sha in this file is itself a tier-crossing decision. The
# commit that changes this file is audited. The unlock path is NEVER
# "revert the branch" — it's "update the pin OR add a pinned.yaml
# acknowledgment." That keeps restart-as-remediation off the table.
#
# ## Lineage with existing patterns
#
# This file mirrors iac/audit/core-update-days.txt (the freeze override
# date list). Both are flat in-tree files the gate consumes. Both treat
# updates as tier-crossing actions. Both are reversible by deletion or
# revert.
#
# ## Known characteristic — the lag-by-1 pattern
#
# Because the pin-update commit is itself a commit, an active-dev
# branch's pin will always be exactly 1 commit behind the tip at the
# end of any session that ends with a pin update. This is expected
# and intentional: the pin marks the "last committed checkpoint," not
# the literal tip. On the NEXT commit after a pin update, the gate
# will BLOCK — that BLOCK is the gate asking "is this new commit
# intentional?" and the answer is to pin again or add a pinned.yaml
# acknowledgment.
#
# For sessions where the tip drifts faster than the human can update
# pins (e.g., a multi-commit session), the pattern is:
#   1. Work accumulates on lifecycle
#   2. At the natural end of a coherent unit (e.g., a feature lands),
#      update this file to the current tip
#   3. Commit the pin update as the session's final artifact
#   4. Accept that the gate will BLOCK briefly on the next commit
#      until the next pin update
#
# A future enhancement (proposed) is to change the lifecycle check
# from "exact match" to "pin must be an ancestor of tip" using
# `git merge-base --is-ancestor`. That would remove the lag-by-1
# characteristic entirely. Not done tonight — scope.
